### 3. Testing Approach

**Question**: 
- Unit tests (Jest/Vitest)?
- Integration tests (API endpoints)?
- Manual testing (Postman/curl)?
- All of the above?

**Answer**:
- Question: What is unit, integration test and manual testing, what's their procedure?
- **Clarification**: See `docs/testing-guide.md` for detailed explanation:
  - **Unit tests**: Test individual functions in isolation (e.g., validation functions)
  - **Integration tests**: Test how multiple components work together (e.g., API endpoint with database)
  - **Manual testing**: Test by hand using Postman/curl to verify user experience
- All three types will be used

### 4. Break Plans Business Logic

**Question**: 
For `/break/plans` POST:
- How should `title` and `description` be generated from `feeling` and `location`?
- Do you need reverse geocoding (coordinates → address)?
- Should plans be stored in Firestore immediately, or is there additional processing?

**Answer**:
- Title and description: Generated by an LLM agent that reads map and finds locations within reach of the customer and makes them feel better under the given mood
- Reverse geocoding: Yes, needed
- Storage: Store all generated plans in Firestore immediately with plan detail and the order it appears in the frontend
- When user selects a plan: The `chosen` field of that plan is set to `True`

## Decisions Made

1. ✅ **Authentication**: Firebase Auth with JWT tokens, support email/password and Google Sign-In
2. ✅ **Testing**: Use all three types (unit, integration, manual)
3. ✅ **Break Plans**: LLM generates plans, reverse geocoding needed, store immediately with order
4. ✅ **History**: Filter by `chosen: true`, default limit 7, pull-to-refresh pagination
5. ✅ **CI/CD**: GitHub Actions for both backend API and iOS app (separate workflows)
6. ✅ **Schema**: Design together, need 4 collections (break_plans, history, users, voice_conversations)
7. ✅ **Implementation Order**: API contract review → Schema design → Auth → Middleware → Endpoints
